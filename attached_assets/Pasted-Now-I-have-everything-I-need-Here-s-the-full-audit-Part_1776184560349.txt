Part 1: The Fix in knowledgeQA.ts — Verified Correct
Commit 6d228c2 shows exactly what changed. Let me walk through what the bug actually was, because the "LangGraph drops arrays" framing is slightly off from what the code reveals:

The Actual Root Cause
The old workflow ran in this order: parseQuestionNode → loadBrandContextNode → retrieveAndScoreNode

Old parseQuestionNode: generated 3 phrase embeddings, stored them in state.questionEmbeddings as number[][] (a 3-element array of 1536-float vectors).

Old loadBrandContextNode (the bug):

if (state.input.brandContext) {
  const compositeText = state.input.brandContext + "\n" + state.input.question;
  const [embedding] = await ... embedDocuments([compositeText]);
  return { brandContext: ..., questionEmbeddings: [embedding] };  // ← 1-element array
}

loadBrandContextNode was OVERWRITING questionEmbeddings with a single-element array. It took the 3 embeddings parseQuestionNode had just generated and replaced them with one composite embedding of brandContext + originalQuestion. The 3 reformulations were silently thrown away.

This wasn't LangGraph truncating large arrays. It was a code-level overwrite — loadBrandContextNode explicitly set questionEmbeddings: [embedding], a length-1 array. The user's intuition about "vecCount: 1 at retrieve" was correct, but the mechanism was cleaner than "state truncation."

The composite embedding was AnswerMonk brand context + customer acquisition question. Since the brand context dominated (it's longer than the question), the embedding drifted toward AEO content. By the time retrieveAndScoreNode ran, it had exactly 1 vector that was mostly about AEO, and vector search returned AEO content.

The Fix — Clean and Correct
The new code:

State annotation changed: questionEmbeddings: Annotation<number[][]> → queryPhrases: Annotation<string[]>. Strings, not float arrays, travel in state. Lightweight and impossible to silently overwrite with a shorter array.
parseQuestionNode: no longer creates embeddings. Just expands the query and stores queryPhrases (3 strings) in state.
loadBrandContextNode: no longer creates embeddings. Just resolves brand context to a string and stores it in state. Returns { brandContext } only.
retrieveAndScoreNode: generates embeddings FRESH at the top of the node:
const phrases = state.queryPhrases.length > 0 ? state.queryPhrases : [state.input.question];
const brandPrefix = state.brandContext ? `${state.brandContext}\n` : "";
const textsToEmbed = phrases.map((q) => `${brandPrefix}${q}`);
const embedResults = await Promise.all(
  textsToEmbed.map((t) => withRetry(() => embedModel.embedDocuments([t]), 2, "embed_retrieve"))
);
const vecs = embedResults.map(([e]) => e);

3 phrases × (brand prefix prepended) = 3 branded texts
Each one gets its own fresh embedding
No state hand-off of float arrays anywhere — the vectors exist only in local scope
Impossible to drop to 1 because there's no node transition between embed and search
The new log line QA:retrieve — embeddings generated with phraseCount + vecCount + hasBrandPrefix + phrases proves whether 3 embeddings actually got generated.

What This Means for the Retrieval Quality
The fix gives you the architecture that was supposed to exist from the start:

RRF actually runs on 3 diverse vectors now, not 1 drifted composite
Brand context is prepended to EACH phrase — so each of the 3 retrievals is brand-aware in the same way, but the 3 phrases remain semantically distinct (original question + 2 reformulations)
RRF merge across 3 lists per table actually happens, surfacing B2B content that was drowned by the drifted single-vector search
The verification you saw in the logs (topPrinciple: "AEO Is a Core B2B GTM Motion in 2025", topPlaybook: "B2B Growth Sales Strategy Implementation", topSimilarity: 0.6637, finalConfidence: 0.90) confirms all 3 things working together.
