Part 2 — ef_search=64: This is a real gap.

I grepped the Sieve codebase earlier and found zero references to ef_search or hnsw.ef_search. Sieve uses the pgvector default of ef_search = 40. Bumping to 64 would improve recall from ~94% to ~98% at minimal latency cost (about 10-20ms extra per query).

Implementation: either

Set globally at DB level: ALTER DATABASE <name> SET hnsw.ef_search = 64
Or set per-query: SET LOCAL hnsw.ef_search = 64 at the start of each search transaction
The per-query approach is better because it scopes the setting to retrieval queries only, not every query hitting the DB.

Part 3 — Multi-query expansion + RRF: This is a meaningful upgrade.

Sieve already has query expansion — parseQuestionNode in knowledgeQA.ts generates 2 reformulations via gpt-4o-mini and averages the 3 vectors together. Averaging is the quick-and-dirty approach; it works but blurs signal.

RRF (Reciprocal Rank Fusion) is strictly better:

Embed each query phrase separately → 3 different vectors
Run 3 separate retrievals → 3 different candidate pools (each with its own ranking)
Merge with RRF: score(doc) = Σ 1 / (k + rank_i) where k is typically 60
Top-N from merged scores become the final candidates
This catches objects that are strongly relevant to ONE sub-query but weakly relevant to the others — those get surfaced by RRF but drowned out by averaging.
